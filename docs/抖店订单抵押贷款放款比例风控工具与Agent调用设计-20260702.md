# 抖店订单抵押贷款放款比例风控工具与 Agent 调用设计

版本：2026-07-02 落地版 v2  
定位：从资深数据分析师 / 风控策略 / 工程落地视角，设计“以抖店订单形成的可回款资产作为放款参考”的计算规则、风控工具、Agent 调用链路和可实施任务拆分。  
适用：可作为独立“融资风控巡检服务”设计，也可落到现有平台；本文暂不绑定具体服务边界。推荐抽象为：风控巡检服务、Agent 报告服务、通知/outbox 模块、原业务系统。

## 0. 来源、假设和不确定项

### 0.1 已核对来源

本设计参考：

1. `产品文档.md`
   - 系统只输出风险参考、证据说明、核实事项和建议动作，不执行放款、停放或调额。
   - 规则引擎是风险事件、风险等级、影响金额、建议动作、通知策略和证据的唯一来源。
   - 面向运营的报告 Agent 只能解释规则引擎输出，不得新增事实、改变等级、改变金额或创造资产计算公式。
   - 风控分析 Agent 可以作为规则引擎内部的判断和配置模块，但必须输出结构化证据、置信度、规则版本和审计日志。
   - 每次巡检必须记录数据快照时间，数据缺失或过期必须提示。
   - 可回款资产口径由规则引擎和字段映射配置，不在产品层固定唯一公式。
2. `风险点来源.md`
   - 第一还款来源是订单形成的净可归集货款。
   - 只有落到订单、售后、履约、结算、平台扣款、回款账户、权属或扣收优先级证据上的风险，才能影响可回款资产金额。
   - 同一金额只扣一次；R18 是平台费用和优先扣款的唯一金额扣减入口。
   - 数据缺失、授权失效、接口延迟、字段缺失或口径冲突不等于确认业务风险。
   - 首期核心自动规则：R03、R05、R06、R13、R15、R16、R17、R18、R19、R20、R21、R30。
3. `docs/电商平台融资自动巡检执行落地方案-20260702.md`
   - 既有平台文档提供规则、任务、审计、outbox、Agent 报告等能力参考。
   - 本文不强依赖具体服务边界；如果重新起独立服务，应保留这些能力，而不是照搬原有系统拆分。

### 0.2 本文业务假设

1. “抖店商家发货到回款大概一个月左右”来自本轮业务描述，本文不把它当成抖店官方固定账期。
2. 系统应把账期写成可配置字段：`settlement_lag_days`。
3. 首期可以先把 `settlement_lag_days` 默认配置为业务确认值，例如约 30 天；该数值必须用真实店铺历史结算单校验。
4. 本文出现的放款比例、评分阈值、风险折减系数均为“模型参数设计”，不是行业统计结论；正式启用前必须用历史订单、退款、结算、回款和坏账样本校准。

### 0.3 不能越过的边界

1. 风控分析 Agent 可以根据抖店原始数据做风险判断、趋势推断和规则配置，但必须在风控引擎内部执行，不能绕过审计。
2. 报告 Agent 不能自己计算可放款金额，也不能根据对话临时改金额。
3. 规则引擎和风控工具输出的是放款参考，不代表系统已经执行放款。
4. 放款审批、放款执行、调额、停放仍在原业务系统和原业务流程完成。
5. 如果只是模型推断刷单、退款趋势或履约风险，报告中必须写成“疑似”“预测”“建议核实”，不能写成平台已确认事实。

### 0.4 数字口径说明

为避免把示例参数误用为正式策略，本文所有数字分为三类：

| 类型 | 说明 | 使用要求 |
| --- | --- | --- |
| 业务口径数字 | 例如本轮业务描述中的“商家发货回款周期大概一个月左右” | 只能作为 `settlement_lag_days` 初始配置候选，必须用真实结算数据校准 |
| 示例数字 | JSON 示例中的金额、比例、时间 | 仅用于说明字段关系，不得上线为默认策略 |
| 规则编号 | R03、R05、R06、R13、R15、R16、R17、R18、R19、R20、R21、R30 | 来源于 `风险点来源.md`，作为首期核心规则编号 |

上线前必须把所有 `cfg` 参数写入配置表，并在规则版本中留痕。

## 1. 业务问题建模

### 1.1 现金流缺口

抖店商家常见资金链路：

```text
进货付款 -> 上架销售 -> 买家下单付款 -> 商家发货 -> 买家签收/售后期 -> 平台结算 -> 商家回款
```

业务痛点：

1. 商家订单已经产生，但平台尚未回款。
2. 商家需要提前进货或补货。
3. 平台可基于“订单形成的净可归集货款”评估可回款资产。
4. 放款比例必须扣除或折减退款、刷单、履约、店铺状态、平台处罚、账户控制、重复融资等风险。

### 1.2 资金巡警模式

这个系统可以理解为“资金巡警”，但不是“自动审批员”。

资金巡警做三件事：

1. 实时巡检：持续读取风控返回的抖店订单、售后、履约、结算、账户、店铺状态等数据。
2. 发现风险：一旦命中退款、刷单、封店、限制提现、平台扣款、回款路径异常、权属冲突等风险，立即生成结构化风险事件。
3. 汇报和调整参考额度：把风险点推送给负责运营，并实时重算“未放款金额中的可继续放款参考额度”。

默认边界：

```text
系统实时调整的是“建议可放款余额 / 未放款额度参考值”，不是直接执行放款、停放或调额。
```

如果业务后续要求系统直接联动原业务系统调整未放款金额，必须升级为“受控执行模式”：

1. 原业务系统提供明确的额度冻结、额度释放、额度调整接口。
2. 每次调整都必须有规则事件、影响金额、操作原因、审批策略和审计记录。
3. 严重风险可自动冻结未放款余额，但需要业务侧明确授权。
4. 普通风险默认只预警和建议调整，不自动执行。
5. 所有自动执行动作都必须支持回滚或人工解除。

### 1.3 风控目标

目标不是“尽量多放款”，而是在可解释、可追溯前提下，找到风险可控的最高参考放款上限：

```text
建议放款上限 = min(商家申请金额, 进货资金需求, 风险调整后可放款资产 - 已有放款敞口)
```

如果任一核心证据缺失，系统应输出“无法估算原因”，而不是用 Agent 猜测。

### 1.4 第一还款来源

第一还款来源只认“订单形成的净可归集货款”：

```text
净可归集货款 =
  可结算订单金额
  - 已退款金额
  - 预计退款金额
  - 平台扣款
  - 佣金/技术服务费/广告等优先扣款
  - 保证金扣罚
  - 结算冻结
  - 权属争议或重复融资冻结
  - 回款路径失控剔除
```

GMV、订单量、经营趋势、达人依赖、利润质量、主体信用可以作为风险信号，但不能直接当成可回款资产。

## 2. 未放款金额实时重算

### 2.1 核心概念

| 字段 | 含义 |
| --- | --- |
| `approved_credit_limit` | 已审批总额度，来自原业务系统 |
| `disbursed_amount` | 已放款金额 |
| `pending_disbursement_amount` | 已审批但尚未放款金额 |
| `risk_adjusted_loan_limit` | 风控工具实时计算的风险调整后建议放款上限 |
| `available_to_disburse_reference` | 当前建议仍可继续放款金额 |
| `suggested_hold_amount` | 建议暂缓/冻结的未放款金额 |

### 2.2 实时计算公式

```text
available_to_disburse_reference =
  max(risk_adjusted_loan_limit - disbursed_amount, 0)

suggested_hold_amount =
  max(pending_disbursement_amount - available_to_disburse_reference, 0)
```

解释：

1. 已放出去的钱不能通过本工具直接收回，只能进入贷后或还款跟踪。
2. 未放出去的钱可以实时重算“建议还能放多少”。
3. 如果风险升高，系统先建议暂缓未放款部分，并通知运营核实。
4. 如果风险解除，系统可以建议恢复部分未放款额度。

### 2.3 巡警触发条件

资金巡警不只按日跑，也需要事件触发：

| 触发事件 | 处理 |
| --- | --- |
| 新订单进入订单池 | 重新计算可回款资产 |
| 订单发货 / 签收 | 更新订单阶段系数 |
| 售后申请 / 退款成功 | 运行 R16，重算扣减或冻结 |
| 平台处罚 / 限制提现 / 店铺封禁 | 运行 R03/R05/R20，必要时建议暂缓未放款 |
| 物流异常 / 拒收 / 退回 | 运行 R15，折减或剔除订单资产 |
| 结算账单更新 | 运行 R17/R18/R19，更新净可归集货款 |
| 回款账户或代扣授权变化 | 运行 R20，核实回款路径 |
| 发现重复融资或权属冲突 | 运行 R21，冻结或剔除相关订单资产 |
| 数据授权失效 / 字段缺失 | 运行 R30，降低置信度或无法估算 |

### 2.4 通知给负责运营

当 `suggested_hold_amount > 0` 或命中高风险/严重风险时，系统应通知对应运营：

```text
【抖店融资巡检】{店铺名} 未放款金额建议调整
风险等级：{risk_level}
建议动作：{suggested_action}
建议暂缓金额：{suggested_hold_amount}
主要原因：{top_risks}
报告链接：{report_url}
```

运营打开报告后必须能看到：

1. 哪些订单仍可作为抵押订单。
2. 哪些订单被扣减、冻结或剔除。
3. 哪些风险导致未放款金额需要暂缓。
4. 需要核实的证据和动作。

## 3. 放款比例核心公式

### 3.1 核心变量

| 变量 | 含义 | 来源 |
| --- | --- | --- |
| `gross_order_pool` | 候选订单池金额 | 抖店订单快照 |
| `eligible_order_asset` | 满足入池条件的订单资产 | 订单入池工具 |
| `deduct_amount` | 确认或可预计扣减金额 | R16、R18、R19 等规则 |
| `freeze_amount` | 风险冻结金额 | R17、R20、R21 等规则 |
| `exclude_amount` | 剔除金额 | R03、R05、R06、R13、R20、R21 |
| `net_collectible_asset` | 净可归集货款 | 资产口径计算器 |
| `existing_exposure` | 已放款敞口 | 放款/还款系统 |
| `safety_buffer` | 安全垫 | 风控配置 |
| `base_advance_rate` | 基础放款比例 | 资产口径配置 |
| `risk_multiplier` | 风险折减系数 | 风控工具聚合 |
| `data_confidence_multiplier` | 数据置信折减系数 | R30 数据质量工具 |
| `final_advance_rate` | 最终参考放款比例 | 规则引擎输出 |
| `recommended_loan_limit` | 建议放款上限 | 规则引擎输出 |
| `available_to_disburse_reference` | 当前建议可继续放款金额 | 规则引擎输出 |
| `suggested_hold_amount` | 建议暂缓/冻结的未放款金额 | 规则引擎输出 |

### 3.2 资产计算公式

```text
eligible_order_asset = sum(order.expected_settlement_amount * order_stage_factor)

net_collectible_asset =
  eligible_order_asset
  - deduct_amount
  - freeze_amount
  - exclude_amount

risk_adjusted_asset =
  max(net_collectible_asset - safety_buffer, 0)

final_advance_rate =
  base_advance_rate
  * risk_multiplier
  * data_confidence_multiplier

recommended_loan_limit =
  min(
    requested_amount,
    purchase_cash_need,
    max(risk_adjusted_asset * final_advance_rate - existing_exposure, 0)
  )

available_to_disburse_reference =
  max(recommended_loan_limit - disbursed_amount, 0)

suggested_hold_amount =
  max(pending_disbursement_amount - available_to_disburse_reference, 0)
```

说明：

1. `base_advance_rate` 不写死，由资产口径配置决定。
2. `risk_multiplier` 由各风控工具输出后聚合。
3. `data_confidence_multiplier` 由 R30 数据授权/数据质量工具输出。
4. `existing_exposure` 不属于可回款资产，但必须参与安全垫和放款上限。
5. 如果 `net_collectible_asset` 无法估算，不能输出确定放款比例，只能输出“无法估算”或人工核实建议。

### 3.3 订单阶段系数

订单不是支付成功就能全额抵押。不同订单阶段对应不同入池方式：

| 订单阶段 | 入池处理 | 风险原因 |
| --- | --- | --- |
| 已付款未发货 | 默认不入池或低比例入池 | 尚未履约，取消和未发货风险高 |
| 已发货未签收 | 延迟或折扣入池 | 物流、拒收、虚假发货风险 |
| 已签收售后期内 | 折扣或冻结部分入池 | 售后退款风险尚未结束 |
| 售后期结束待结算 | 较高比例入池 | 可回款确定性较高，但仍有平台扣款和冻结 |
| 已结算未到账 | 较高比例入池 | 重点看回款账户、扣收优先级和冻结 |
| 已退款/取消/无效 | 剔除 | 资产已灭失 |

每个阶段的比例不在本文写死，落地时配置为：

```json
{
  "paid_unshipped_rate": "cfg",
  "shipped_unsigned_rate": "cfg",
  "signed_after_sale_open_rate": "cfg",
  "after_sale_closed_rate": "cfg",
  "settled_pending_payout_rate": "cfg"
}
```

## 4. 风控工具总体设计

### 4.1 工具分工

这里的“工具”不要求一开始就有外部接口或成熟模型。首期可以先把工具实现为内部判断模块：特征计算、规则判断、LLM 风控分析、参数配置、聚合计算。  
关键不是有没有现成工具，而是每个判断都必须结构化、可复跑、可审计。

风控分析 Agent 的定位：

1. 可以读取原始订单、售后、履约、结算、账户等结构化数据。
2. 可以计算当前商品退款率、店铺退款率、异常订单聚集度、履约异常趋势等特征。
3. 可以基于特征判断疑似刷单、未来退款风险、回款路径风险、履约风险。
4. 可以生成或更新风控引擎配置草案，例如特征权重、规则阈值、提示词版本、风险等级映射。
5. 不能直接执行放款、停放或调额；最终动作仍输出为建议和待核实事项。

| 工具 | 对应规则 | 核心输出 |
| --- | --- | --- |
| `risk_feature_calculator` | 所有规则前置 | 当前退款率、预测退款率、异常聚集度、履约异常、回款控制等特征 |
| `llm_risk_judgement_agent` | R03/R05/R06/R13/R15/R16/R17/R18/R19/R20/R21/R30 | Agent 风控判断、风险等级、证据链、置信度、建议动作 |
| `policy_config_agent` | 策略配置 | 生成规则配置、特征权重、风险等级映射、回测任务 |
| `order_pool_eligibility_tool` | R13、R15、R16、R17 | 哪些订单可入池、打折、冻结、剔除 |
| `net_settlement_asset_tool` | R17、R18、R19 | 净可归集货款、平台扣款、冻结、保证金影响 |
| `refund_risk_tool` | R16 | 已退款、预计退款、售后未完结冻结、退款率异常 |
| `fake_order_risk_tool` | R13、R06 | 刷单/虚假交易/欺诈发货订单剔除或降比例 |
| `shop_status_risk_tool` | R03、R05 | 店铺封禁、限制提现、平台处罚、清退、风控冻结 |
| `fulfillment_risk_tool` | R15 | 未发货、延迟、拒收、物流异常、无法履约 |
| `account_control_tool` | R20 | 回款账户、监管账户、代扣授权、回款路径失控 |
| `receivable_ownership_tool` | R21 | 权属不清、重复融资、优先扣收冲突 |
| `data_quality_tool` | R30 | 授权、接口、字段、新鲜度、无法估算原因 |
| `loan_ratio_aggregator_tool` | 聚合工具 | 最终放款比例、建议上限、风险解释、核实事项 |

```json
    "核实未完结售后是否继续扩大"
  ]
}
```

## 5. 核心风控工具细化

### 5.1 订单池入池工具

工具名：`order_pool_eligibility_tool`

目标：先判断订单能不能作为抵押订单池，不能直接用 GMV。

输入字段：

1. 订单 ID、支付金额、应结算金额。
2. 支付时间、发货时间、签收时间、预计结算时间。
3. 订单状态、售后状态、物流状态。
4. 买家维度、收货地址、商品/SKU。
5. 订单是否已被其他融资占用。

输出：

| 输出 | 说明 |
| --- | --- |
| `include_amount` | 正常入池金额 |
| `discount_amount` | 打折入池金额 |
| `freeze_amount` | 暂时冻结金额 |
| `exclude_amount` | 剔除金额 |
| `order_stage_factor` | 订单阶段折算系数 |
| `evidence` | 订单状态、时间、售后、物流证据 |

硬规则：

1. 已退款、取消、平台判定无效的订单剔除。
2. 权属不清或重复融资的订单转 R21，冻结或剔除。
3. 回款路径不可控的订单转 R20，冻结或剔除。
4. 数据缺失时不直接判定刷单或退款，只输出数据不足。

### 5.2 退款风险工具

工具名：`refund_risk_tool`

对应规则：R16 售后退款风险。

目标：扣除已确定退款，冻结或折减预计退款风险。

关键指标：

1. 已退款金额。
2. 退款中金额。
3. 售后未完结金额。
4. 退款订单数 / 订单数。
5. 退款金额 / 订单金额。
6. 核心商品退款集中度。
7. 近期退款变化趋势。

金额处理：

```text
已退款成功 -> deduct 或 exclude
售后未完结 -> freeze 或 discount
预计退款 -> deduct 或 freeze，按资产口径配置
缺售后数据 -> R30 数据不足，不直接按高退款处理
```

运营解释：

1. 告诉运营是哪批订单、哪个商品、哪个售后原因影响资产。
2. 展示已退款和未完结售后分开计算。
3. 明确影响的是“可放款参考资产”，不是系统已经拒贷。

### 5.3 刷单/虚假交易风险工具

工具名：`fake_order_risk_tool`

对应规则：R13 订单真实性风险、R06 虚假发货与欺诈发货风险。

目标：降低刷单、虚假交易、欺诈发货导致放款后无法回款的风险。

可用信号：

1. 买家集中度异常。
2. 收货地址/手机号/设备/支付账号异常聚集。
3. 同 SKU 短时异常下单。
4. 支付后快速退款或批量退款。
5. 物流轨迹缺失、虚假揽收、异常签收。
6. 平台判定虚假交易、欺诈发货或无效订单。

金额处理：

```text
平台判定无效/欺诈 -> exclude
强疑似且证据充分 -> freeze 或 high risk discount
只有经营异常信号但无订单证据 -> 辅助标记，不直接扣减
数据缺失 -> R30 数据不足
```

硬停条件：

1. 平台已判定批量虚假交易。
2. 店铺核心订单池大比例被平台否定。
3. 相关订单回款路径已经不可控。

硬停是建议动作 `pause_new`，不是系统执行停放。

### 5.4 店铺状态和平台处罚工具

工具名：`shop_status_risk_tool`

对应规则：R03 店铺状态风险、R05 平台违规风险。

目标：识别店铺封禁、限制提现、风控冻结、清退、平台处罚等导致订单无法归集或无法结算的风险。

关键字段：

1. 店铺经营状态。
2. 是否限制提现。
3. 是否限制发布商品或发货。
4. 是否平台处罚、罚款、赔付、违约金。
5. 是否临时风控冻结。
6. 是否主体认证失效或清退。

处理：

| 场景 | 处理 |
| --- | --- |
| 限制提现 | 转 R20 或 R17，冻结相关资产 |
| 清退/封店 | 高风险或严重风险，相关订单冻结或剔除 |
| 平台罚款/赔付 | 转 R18 扣减 |
| 临时冻结 | R17/R05 冻结 |
| 主体认证失效 | 转 R20/R21 核实回款权 |

### 5.5 履约风险工具

工具名：`fulfillment_risk_tool`

对应规则：R15 订单履约风险。

目标：识别订单无法变成可结算资产的风险。

关键字段：

1. 发货状态。
2. 物流轨迹。
3. 签收状态。
4. 延迟天数。
5. 拒收、退回、异常件。
6. 平台履约处罚。

处理：

```text
未发货 -> 不入池或低比例入池
延迟但仍可履约 -> 后移预计结算日或折扣
无法履约/拒收/退回 -> deduct 或 exclude
履约异常导致赔付 -> R18 扣减
```

### 5.6 结算和平台扣款工具

工具名：`net_settlement_asset_tool`

对应规则：R17 结算周期风险、R18 平台扣款风险、R19 保证金风险。

目标：计算平台实际会先扣什么，再回什么。

关键字段：

1. 预计结算日。
2. 账期变化。
3. 结算冻结。
4. 平台佣金。
5. 技术服务费。
6. 违约金、赔付。
7. 达人佣金、广告费等优先扣款。
8. 保证金补缴或扣罚。

处理：

1. R18 是平台费用和优先扣款唯一入口。
2. R23、R25、R27 只引用 R18，不重复扣减。
3. 保证金如果已作为平台扣款进入 R18，R19 不再重复扣减。
4. 账期延长不一定扣金额，但会影响现金流和安全垫。

### 5.7 回款路径控制工具

工具名：`account_control_tool`

对应规则：R20 回款路径控制风险。

目标：确认订单应收款最终能不能进入受控账户并用于扣收。

关键字段：

1. 回款账户。
2. 监管账户。
3. 代扣授权。
4. 账户变更记录。
5. 司法冻结或账户冻结。
6. 平台官方金融优先扣收。

处理：

| 场景 | 处理 |
| --- | --- |
| 回款账户受控且授权有效 | 正常 |
| 账户变更但可核实 | manual_check |
| 代扣授权失效 | freeze 或 pause_new 建议 |
| 回款不进受控路径 | exclude 或 pause_new 建议 |
| 只有账户数据缺失 | R30 数据不足 |

### 5.8 权属和重复融资工具

工具名：`receivable_ownership_tool`

对应规则：R21 应收款权属与重复融资风险。

目标：确认同一批订单应收款没有被重复融资，且我方具备优先回收权。

关键字段：

1. 订单/应收款是否已质押。
2. 融资合同。
3. 其他资金方占用。
4. 平台官方金融扣款优先级。
5. 回款账户绑定和扣收优先级。

处理：

1. 同一订单已被重复融资：冻结或剔除。
2. 权属不清：冻结并人工核实。
3. 优先扣收冲突：冻结或降低放款比例。
4. 复杂供应链确权不足：首期不自动放大额度，只作为核实事项。

### 5.9 数据质量工具

工具名：`data_quality_tool`

对应规则：R30 数据授权风险。

目标：防止“看不见风险”被误认为“没有风险”。

检查：

1. 订单数据是否可见。
2. 售后数据是否可见。
3. 物流数据是否可见。
4. 结算账单是否可见。
5. 回款账户是否可见。
6. 数据时间是否过期。
7. 字段是否缺失。
8. 授权是否失效。

输出：

```text
数据完整 -> data_confidence_multiplier = cfg.full
非核心字段缺失 -> data_confidence_multiplier = cfg.partial
核心字段缺失 -> 无法估算或 pause_new 建议
授权失效 -> R30 数据不足；若账户/代扣业务证据失效，再转 R20
```

## 6. 放款比例聚合策略

### 6.1 聚合顺序

```text
第一步：数据质量检查
第二步：订单入池与阶段折算
第三步：扣减项计算
第四步：冻结项计算
第五步：剔除项计算
第六步：风险折减系数计算
第七步：硬停规则判断
第八步：输出最终放款比例和上限
```

### 6.2 硬停规则

任一条件满足时，不输出确定放款比例，只输出人工核实或暂停新增放款参考：

1. 核心订单、售后、结算、回款账户数据不可见，且无法替代验证。
2. 店铺被封、清退或严重限制提现。
3. 回款路径不受控或代扣授权失效。
4. 核心订单池权属不清或重复融资。
5. 平台已判定批量虚假交易或欺诈发货。
6. 资产口径配置冲突，无法计算净可归集货款。

### 6.3 风险折减聚合

聚合原则：保守取低，不做平均掩盖高风险。

```text
risk_multiplier = min(
  refund_multiplier,
  fake_order_multiplier,
  shop_status_multiplier,
  fulfillment_multiplier,
  settlement_multiplier,
  account_control_multiplier,
  ownership_multiplier
)
```

如果业务希望更精细，可以在历史数据充分后改为分层模型：

```text
expected_loss_rate =
  refund_expected_loss_rate
  + fraud_expected_loss_rate
  + fulfillment_expected_loss_rate
  + platform_deduction_expected_loss_rate
  + account_control_expected_loss_rate
  + ownership_expected_loss_rate

final_advance_rate =
  base_advance_rate * max(1 - expected_loss_rate - safety_margin_rate, 0)
```

说明：第二种期望损失模型需要历史回款、坏账、退款、冻结和处罚样本，不建议第一天就上线为自动决策。

## 7. Agent 调用链路

### 7.1 Agent 分层

这里必须区分两个 Agent，不能混在一起：

| Agent | 定位 | 能力 | 边界 |
| --- | --- | --- | --- |
| 风控分析 Agent | 风控引擎内部的判断和配置主体 | 读取结构化原始数据，计算特征，判断疑似风险，预测退款趋势，生成规则配置 | 必须输出证据、置信度、版本和审计；不能直接执行放款 |
| 报告 Agent | 面向运营的解释和沟通主体 | 把风控分析结果翻译成运营报告、群聊提醒、核实清单 | 不能重新计算金额，不能把模型推断说成平台确认事实 |

因此，本文后续说“Agent 有风控判断”时，指的是“风控分析 Agent”；说“Agent 不能改金额”时，指的是“报告 Agent 不能绕过风控引擎”。

### 7.2 正确链路

```text
运营/系统触发巡检
  -> 业务入口或 Agent 报告服务收集上下文并触发任务
  -> 风控巡检服务 financing.inspection.create / rerun
  -> 风控巡检服务生成数据快照
  -> risk_feature_calculator 计算退款率、履约异常、订单聚集、回款路径等特征
  -> 风控分析 Agent 读取特征和原始证据，判断疑似刷单、未来退款风险、履约风险等
  -> policy_config_agent 生成或选择当前规则配置
  -> loan_ratio_aggregator_tool 聚合金额、风险等级和建议动作
  -> 风控巡检服务保存风险事件、金额台账、报告输入和审计
  -> Agent 报告服务读取结构化结果
  -> Agent 报告服务生成运营可读报告和解释
  -> 通知/outbox 模块发送通知
  -> 运营查看报告、核实风险、反馈
```

### 7.3 风控分析 Agent 可以做什么

1. 计算当前商品退款率、店铺退款率、订单阶段退款率。
2. 用当前退款表现、历史退款表现、商品类目表现和店铺表现，预测未来退款风险。
3. 识别疑似刷单或异常交易，例如买家聚集、地址聚集、短时爆发、批量退款、物流异常组合。
4. 判断是否存在履约风险、回款路径失控、平台处罚风险、权属冲突风险。
5. 输出风险等级、证据链、置信度、影响金额、建议动作。
6. 生成风控引擎配置，例如特征权重、等级映射、策略版本、提示词版本。
7. 对历史样本做回测，总结哪些规则有效、哪些规则误报或漏报。

### 7.4 报告 Agent 可以做什么

1. 选择需要展示的风险顺序。
2. 把风控分析结果翻译成运营能看懂的话。
3. 解释为什么某些订单被扣减、冻结或剔除。
4. 生成核实事项。
5. 生成通知摘要。
6. 回答“这笔放款为什么只能参考这个比例”。

### 7.5 风控分析 Agent 不能做什么

1. 不能把没有证据的数据补出来。
2. 不能把“疑似刷单”写成“平台已确认刷单”，除非数据源明确给出平台确认结果。
3. 不能在没有版本记录的情况下修改规则配置。
4. 不能绕过金额台账直接改建议放款上限。
5. 不能把缺失售后、结算、账户数据当成低风险。
6. 不能直接执行放款、停放、调额。

### 7.6 报告 Agent 不能做什么

1. 不能重新计算退款率。
2. 不能重新判断刷单。
3. 不能改风控分析 Agent 输出的风险等级。
4. 不能改聚合工具输出的建议金额。
5. 不能把建议动作写成已执行动作。

### 7.7 风控分析 Agent 判断输出规范

风控分析 Agent 每次判断都必须输出结构化结果：

```json

  "snapshot_id": "snap_20260702_001",
  "rule_id": "R13",

  "risk_level": "high",
  "confidence": "model_calculated",
  "judgement_type": "suspected_not_confirmed",
  "impact_type": "freeze",
  "impact_amount": "calculated_by_engine",

    {
      "feature": "buyer_concentration",
```

### 7.8 报告 Agent 表达规范

报告 Agent 必须按风控分析结果表达：

```text
根据风控分析 Agent 和聚合工具输出，本次可回款资产主要受 {top_risks} 影响。
其中 {risk_id} 属于模型疑似判断，不是平台已确认事实，需要运营核实。
本次建议放款上限由风控引擎按资产口径版本 {版本} 计算，建议动作是 {建议动作}。
需要运营重点核实：{核实事项}。
```

### 7.9 商品退款率预测

用户提到的“一个商品当前退款率，用来推断未来商品退款率”，应作为 R16 的核心预测能力。

#### 7.9.1 特征定义

| 特征 | 含义 | 用途 |
| --- | --- | --- |
| `product_current_refund_rate_by_amount` | 当前观察窗口内，商品退款金额 / 商品支付金额 | 判断当前退款压力 |
| `product_current_refund_rate_by_count` | 当前观察窗口内，商品退款订单数 / 商品订单数 | 判断退款订单占比 |
| `product_pending_after_sale_rate` | 售后中金额 / 商品支付金额 | 预测未来退款释放 |
| `product_historical_refund_rate` | 商品历史退款率 | 平滑当前波动 |
| `store_historical_refund_rate` | 店铺历史退款率 | 商品样本不足时补充 |
| `category_baseline_refund_rate` | 类目基准退款率 | 仅在有可信数据源时使用 |
| `new_product_flag` | 是否新品 | 新品样本不足时降低置信 |
| `promotion_flag` | 是否活动或大促 | 判断退款率短期偏移 |

注意：`category_baseline_refund_rate` 如果没有自有可信数据，不得凭经验捏造。

#### 7.9.2 预测方法

首期不需要复杂黑盒模型，可以用可解释的平滑预测：

```text
predicted_product_refund_rate =
  weighted_smooth(
    product_current_refund_rate,
    product_historical_refund_rate,
    store_historical_refund_rate,
    category_baseline_refund_rate
  )
  + pending_after_sale_adjustment
  + promotion_adjustment
  + data_confidence_adjustment
```

```json
    "category_baseline_weight": "cfg",
    "pending_after_sale_adjustment": "cfg",

```

风控分析 Agent 的职责：

1. 判断当前商品退款率是否异常。
2. 判断样本是否足够支撑预测。
3. 判断未完结售后是否会继续释放为退款。
4. 输出预测退款率、置信度、证据和核实事项。

输出示例：

```json
  ]

```

### 7.10 疑似刷单判断

如果没有外部反欺诈工具，风控分析 Agent 仍然要能基于订单证据做“疑似刷单”判断。  
但判断必须分层：平台确认、模型疑似、证据不足。

#### 7.10.1 可用特征

| 特征 | 说明 |
| --- | --- |
| `buyer_concentration` | 买家集中度是否异常 |
| `receiver_concentration` | 收货地址、手机号、设备等脱敏标识是否聚集 |
| `order_burst_score` | 短时间订单爆发程度 |
| `same_sku_burst_score` | 同 SKU 是否短时集中下单 |
| `rapid_refund_score` | 下单后短时间内退款或售后是否集中 |
| `logistics_abnormal_score` | 物流无轨迹、重复物流、异常签收、虚假揽收 |
| `price_abnormal_score` | 成交价格是否异常偏离常态 |
| `new_buyer_ratio` | 新买家占比是否异常 |
| `platform_fraud_flag` | 平台是否已有虚假交易、欺诈发货、无效订单判定 |

#### 7.10.2 判断原则

1. 有 `platform_fraud_flag` 时，可按平台确认风险处理。
2. 没有平台确认但多项特征同时异常时，输出模型疑似风险。
3. 只有单一弱特征异常时，输出关注或人工核实，不直接剔除资产。
4. 关键字段缺失时，输出 R30 数据质量风险，不用缺失字段“脑补刷单”。
5. 涉及订单金额影响时，必须列出具体订单或订单集合。

#### 7.10.3 输出动作

| 判断类型 | 动作 |
| --- | --- |
| 平台确认无效订单 | exclude |
| 模型强疑似且证据充分 | freeze 或 discount |
| 模型中疑似 | manual_check 或 watch |
| 证据不足 | info，不直接扣减 |
| 数据缺失 | R30，补数据或补授权 |

### 7.11 大模型配置风控引擎

如果风控引擎不开放给普通用户操作，可以由大模型负责配置，但要走“模型生成配置 -> 回测验证 -> 版本发布 -> 审计留痕”的闭环。

#### 7.11.1 配置内容

风控分析 Agent / 策略配置 Agent 可以配置：

1. 特征清单：哪些字段进入退款、刷单、履约、回款路径判断。
2. 特征计算窗口：当前窗口、历史窗口、商品窗口、店铺窗口。
3. 风险等级映射：什么特征组合对应关注、高风险、严重风险。
4. 金额影响方式：deduct、freeze、exclude、discount、info。
5. 报告模板：风险解释、核实事项、通知模板。
6. 回测任务：用历史样本验证误报、漏报和回款覆盖。

#### 7.11.2 配置发布流程

```text
Agent 生成配置草案
  -> 系统保存为 draft_policy_version
  -> 回测模块跑历史样本
  -> 输出回测报告和风险说明
  -> 达到发布条件后发布 active_policy_version
  -> 新巡检任务使用 active_policy_version
  -> 所有结果保留策略版本和提示词版本
```

是否需要人工审批取决于业务授权：

1. 如果是灰度或测试环境，可以允许 Agent 自动发布。
2. 如果是生产环境并影响放款建议，建议至少保留风控负责人审批或自动发布后的可回滚机制。
3. 如果未来要直接联动额度冻结或释放，必须有更强审批、白名单和回滚。

## 8. 运营可解释报告结构

### 8.1 报告必须让运营看懂什么

1. 商家为什么需要钱：订单已产生但平台未回款，需要采购/补货现金。
2. 我们为什么可以参考放款：存在订单形成的净可归集货款。
3. 为什么不能全额放：售后、刷单、平台处罚、履约、回款路径、权属等会影响回款。
4. 本次哪些规则影响最大。
5. 哪些金额被扣减，哪些金额被冻结，哪些订单被剔除。
6. 哪些数据缺失，导致结论不确定。
7. 运营需要核实什么。

### 8.2 报告模板

```text
一、结论
- 风险等级：{risk_level}
- 建议动作：{suggested_action}
- 建议放款上限：{recommended_loan_limit 或 无法估算}
- 最终参考放款比例：{final_advance_rate 或 无法估算}

二、资产口径
- 订单池金额：{gross_order_pool}
- 入池订单资产：{eligible_order_asset}
- 扣减金额：{deduct_amount}
- 冻结金额：{freeze_amount}
- 剔除金额：{exclude_amount}
- 净可归集货款：{net_collectible_asset}
- 已放款敞口：{existing_exposure}
- 资产口径版本：{asset_policy_version}

三、主要风险
1. {rule_id} {rule_name}
   - 影响金额：{impact_amount}
   - 影响方式：{impact_type}
   - 证据：{evidence}
   - 核实事项：{verification_items}

四、放款比例计算链路
- 基础比例：来自资产口径配置
- 风险折减：来自 {top_risks}
- 数据置信折减：来自 R30 数据质量
- 最终比例：由规则引擎输出

五、数据缺失或无法估算
- {missing_fields}
- {unable_to_estimate_reason}

六、运营核实事项
- {verification_items}
```

## 9. 历史数据校准方法

### 9.1 需要沉淀的数据

1. 每日订单池快照。
2. 实际退款结果。
3. 实际平台扣款。
4. 实际结算日和到账日。
5. 实际回款账户。
6. 放款金额、还款结果、逾期结果。
7. 店铺处罚、封禁、冻结、限制提现记录。
8. 运营核实结果。

### 9.2 回测口径

每次回测要回答：

1. 如果当时按该规则放款，最终是否能覆盖本金。
2. 哪些规则提前识别了风险。
3. 哪些规则误报。
4. 哪些风险漏掉了。
5. 哪些参数导致放款比例偏高或偏低。

### 9.3 参数校准

需要校准的参数：

1. `settlement_lag_days`。
2. 各订单阶段 `order_stage_factor`。
3. 各风险工具 `advance_rate_multiplier`。
4. 数据缺失时 `data_confidence_multiplier`。
5. 各等级阈值。
6. `safety_buffer`。
7. `base_advance_rate`。

校准原则：

1. 优先保证回款覆盖安全，再优化放款额度。
2. 高风险样本宁可先人工核实，不让模型自动放大额度。
3. 参数变更必须版本化，并保留回测结果。

## 10. 首期落地建议

### 10.1 第一条闭环

先跑最小闭环：

1. 输入一个抖店订单池快照。
2. 计算订单阶段入池金额。
3. 运行 R16 退款风险工具。
4. 运行 R18 平台扣款工具。
5. 运行 R30 数据质量工具。
6. 输出净可归集货款。
7. 输出建议放款比例和建议上限。
8. Agent 生成运营解释报告。

### 10.2 第二批扩展

再加入：

1. R13/R06 刷单与欺诈发货。
2. R03/R05 店铺状态和平台处罚。
3. R15 履约风险。
4. R17 结算周期风险。
5. R20 回款路径控制。
6. R21 权属和重复融资。
7. R19 保证金风险。

### 10.3 首期不建议做

1. 不建议一开始用复杂黑盒模型直接给放款比例。
2. 不建议让未受控的报告 Agent 读取原始订单后临时判断风险；风控分析 Agent 可以判断，但必须在风控引擎内输出结构化证据和审计。
3. 不建议没有历史回款数据就写死固定放款比例。
4. 不建议把 GMV 或订单量直接当成可抵押资产。
5. 不建议在数据缺失时默认通过。

## 11. 完成定义

满足以下条件，才算“抖店订单抵押贷款放款比例风控工具”首期可用：

1. 订单池能按阶段入池、打折、冻结、剔除。
2. 已退款、预计退款、平台扣款、保证金扣罚能进入金额台账。
3. R18 是平台扣款唯一 owner deduct。
4. 刷单、封店、履约、回款路径、权属、数据质量都有工具输出。
5. 工具输出风险等级、影响金额、建议动作、证据和核实事项。
6. 聚合工具能输出最终参考放款比例和建议放款上限。
7. 风控分析 Agent 能做退款趋势和疑似刷单判断，并保留证据、置信度、策略版本、提示词版本。
8. 报告 Agent 只解释风控结果，不改金额和比例。
9. 运营能看到每个风险点如何影响放款比例。
10. 数据缺失时能明确提示无法估算原因。
11. 所有计算结果、参数版本、工具输出和 Agent 报告都有审计留痕。

## 12. 落地实施蓝图

### 12.1 服务职责划分

| 系统 | 职责 | 禁止事项 |
| --- | --- | --- |
| 风控巡检服务 | 数据快照、特征计算、风控分析 Agent 判断、规则执行、工具编排、金额台账、参数版本、审计、巡检结果输出 | 不让报告 Agent 绕过规则引擎计算，不直接执行放款 |
| Agent 报告服务 | 业务智能体、运营解释报告、群聊/站内通知文案、人工核实流转辅助 | 不直接计算放款比例，不直接修改金额台账 |
| 通知/outbox 模块 | 发送群聊、站内信、待办、Webhook，并记录通知状态 | 不生成新的风险事实，不改变建议动作 |
| 原业务系统 | 授信、放款、还款、额度冻结或释放、审批流 | 不把无审计的人工口径写回规则结果 |
| 风控数据源 | 抖店订单、售后、物流、结算、账户、店铺状态、处罚、授权数据 | 不承担最终解释输出 |

首期工程目标是先让风控巡检服务产出一份“可审计、可复跑、可解释”的结构化巡检结果，再由 Agent 报告服务生成运营报告。  
如果后续重新起服务，只要保留上述职责和审计链路即可，不需要依赖既有系统边界。

### 12.2 一次巡检的完整链路

```text
1. 任务创建
   -> 运营、定时任务或事件触发 `financing.inspection.create`

2. 数据快照
   -> 拉取并固化订单、售后、履约、结算、费用、账户、店铺状态、融资敞口

3. 数据质量检查
   -> `data_quality_tool` 判断核心字段是否完整、授权是否有效、数据是否过期

4. 订单入池
   -> `order_pool_eligibility_tool` 生成订单级资产明细

5. 规则工具执行
   -> R16、R18、R30 首期先跑；后续补 R03/R05/R06/R13/R15/R17/R19/R20/R21

6. 金额台账归集
   -> 把 deduct、freeze、exclude、discount 写成不可重复扣减的 ledger

7. 聚合计算
   -> `loan_ratio_aggregator_tool` 输出建议放款上限、建议可继续放款金额、建议暂缓金额

8. 报告输入固化
   -> 保存报告 Agent 只能读取的结构化 report_input

9. Agent 生成报告
   -> Agent 报告服务按 report_input 解释风险，不改金额、不改等级

10. 通知运营
    -> 通知/outbox 模块发送群聊、站内信或待办

11. 人工核实
    -> 运营填写核实结果，必要时触发复跑

12. 审计归档
    -> 保存快照版本、规则版本、参数版本、工具输出、报告内容和通知记录
```

### 12.3 首期最小闭环

首期不要一次性做全量复杂策略，先把最容易影响金额的闭环跑通：

| 步骤 | 做什么 | 怎么做 | 验收标准 |
| --- | --- | --- | --- |
| 1 | 接入订单快照 | 建立 `fin_data_snapshots`，支持导入或接口写入 | 同一店铺同一时间快照可复查 |
| 2 | 建订单级资产明细 | 建立 `fin_order_asset_items` | 每个订单有入池、冻结、剔除或折扣状态 |
| 3 | 跑 R16 售后退款 | 实现 `refund_risk_tool` | 已退款金额进入 deduct，未完结售后进入 freeze 或配置处理 |
| 4 | 跑 R18 平台扣款 | 实现 `net_settlement_asset_tool` 中的费用扣减 | 平台扣款只在 R18 扣一次 |
| 5 | 跑 R30 数据质量 | 实现 `data_quality_tool` | 核心字段缺失时不能输出确定比例 |
| 6 | 聚合金额 | 实现 `loan_ratio_aggregator_tool` | 能输出净可归集货款和建议动作 |
| 7 | 生成报告输入 | 保存 `fin_report_inputs` | 报告 Agent 只读结构化结果 |
| 8 | 通知运营 | 写入 `fin_notification_records` | 高风险和建议暂缓金额能通知负责人 |

## 17. 规则到金额台账映射

| 规则 | 风险点 | 金额处理 | 是否可硬停 | 证据来源 | 运营动作 |
| --- | --- | --- | --- | --- | --- |
| R03 | 店铺状态异常 | freeze/exclude | 是 | 店铺状态、封禁、清退、限制经营 | 核实店铺是否可经营和可结算 |
| R05 | 平台违规处罚 | deduct/freeze | 是 | 处罚、赔付、限制提现 | 核实处罚金额和解除条件 |
| R06 | 虚假发货/欺诈发货 | exclude/freeze | 是 | 平台判定、物流异常、订单状态 | 核实真实履约证据 |
| R13 | 订单真实性 | freeze/exclude/discount | 是 | 买家聚集、地址聚集、异常退款、平台判定 | 核实订单真实性 |
| R15 | 履约风险 | discount/freeze/exclude | 否，严重时是 | 发货、签收、拒收、退回 | 核实履约进度 |
| R16 | 售后退款 | deduct/freeze | 否，严重时是 | 售后单、退款单 | 核实未完结售后 |
| R17 | 结算周期 | freeze/discount | 否 | 结算账单、预计到账 | 核实结算延迟原因 |
| R18 | 平台扣款 | deduct | 否，严重时是 | 佣金、服务费、赔付、优先扣款 | 核实扣款是否已覆盖 |
| R19 | 保证金 | deduct/freeze | 否 | 保证金扣罚、补缴 | 核实保证金状态 |
| R20 | 回款路径控制 | freeze/exclude | 是 | 回款账户、代扣授权、账户冻结 | 核实监管账户和授权 |
| R21 | 权属/重复融资 | freeze/exclude | 是 | 融资合同、占用订单、优先扣收 | 核实应收款权属 |
| R30 | 数据质量 | info/freeze/unable_to_estimate | 是 | 授权、接口、字段、数据时间 | 补授权或补数据 |

金额处理原则：

1. `deduct` 代表确认从资产中扣掉。
2. `freeze` 代表暂不计入可继续放款资产，但可以人工核实后释放。
3. `exclude` 代表该订单或金额不进入资产池。
4. `discount` 代表按阶段或风险系数折算。
5. `info` 只影响解释和核实，不直接扣减金额。

## 19. 巡警实时调整未放款金额流程

### 19.1 事件流

```text
风控数据变化
  -> 创建或复跑巡检任务
  -> 计算 latest-loan-ratio
  -> 与上一版结果比较
  -> 如果建议暂缓金额增加，生成风险提醒
  -> 如果风险解除，生成恢复建议
  -> outbox 通知运营
  -> 运营核实
  -> 必要时原业务系统执行额度冻结、释放或审批
```

### 19.2 与上一版结果比较

每次巡检后要保存 delta：

| 字段 | 说明 |
| --- | --- |
| `previous_result_id` | 上一次结果 |
| `current_result_id` | 当前结果 |
| `risk_level_changed` | 风险等级是否变化 |
| `suggested_action_changed` | 建议动作是否变化 |
| `recommended_loan_limit_delta` | 建议上限变化 |
| `available_to_disburse_delta` | 建议可继续放款变化 |
| `suggested_hold_amount_delta` | 建议暂缓金额变化 |
| `new_top_risks` | 新增主要风险 |
| `resolved_risks` | 已解除风险 |

### 19.3 通知触发规则

| 条件 | 通知类型 | 接收人 |
| --- | --- | --- |
| `hard_stop = true` | 高优先级风险提醒 | 负责运营、风控负责人 |
| `suggested_hold_amount` 增加 | 未放款暂缓建议 | 负责运营 |
| R30 核心数据缺失 | 数据授权/质量提醒 | 运营、技术支持 |
| 风险解除且可继续放款参考恢复 | 恢复建议 | 负责运营 |
| 工具执行失败 | 系统异常提醒 | 技术支持 |

通知只能表达“建议”和“需核实”，不能表达“已自动放款/已自动停放”，除非后续接入原业务系统的受控执行接口并完成审批授权。

## 21. 验证和测试用例

### 21.1 单元测试

| 用例 | 输入 | 预期 |
| --- | --- | --- |
| 已退款订单 | 订单已退款成功 | 订单不入池或金额 deduct |
| 售后未完结 | 售后申请中 | 金额 freeze，报告提示核实 |
| 平台扣款 | 有服务费和赔付 | R18 deduct，一次扣减 |
| 保证金重复 | 保证金已进 R18 | R19 不重复扣 |
| 核心字段缺失 | 无售后数据 | R30 输出无法确认，不默认无退款 |
| 回款路径失控 | 代扣授权失效 | R20 hard_stop 或 pause_new 建议 |
| 权属冲突 | 订单已被其他融资占用 | R21 freeze/exclude |
| 工具失败 | 结算工具异常 | 聚合结果标记 partial 或 unable_to_estimate |

### 21.2 集成测试

| 场景 | 验证点 |
| --- | --- |
| 完整低风险快照 | 能输出建议上限、报告和通知记录 |
| 高退款快照 | R16 影响金额进入台账，建议暂缓金额变化 |
| 店铺限制提现 | R20/R03 触发硬停或暂停新增建议 |
| 数据授权失效 | R30 触发，Agent 报告不能给确定比例 |
| 人工核实后复跑 | 新结果保留历史，不覆盖旧审计 |
| 风控分析 Agent 判断 | 能输出疑似刷单、退款预测、证据和置信度 |
| 报告 Agent 对话查询 | 只解释已固化风控结果，不生成新金额 |

### 21.3 回归测试

每次修改规则版本或资产口径版本，必须跑：

1. 金额去重测试。
2. 硬停规则测试。
3. 数据缺失测试。
4. 报告 Agent 禁止改金额测试。
5. 报告字段完整性测试。
6. outbox 通知幂等测试。
7. 风控分析 Agent 判断审计测试。
8. 策略配置版本回滚测试。

## 22. 需要业务确认的事项

以下事项我不能替业务直接定，需要业务、风控和产品确认：

1. 首期 `settlement_lag_days` 的默认配置值，以及是否按类目、店铺等级、结算方式分层。
2. 订单阶段系数是否先按统一配置，还是按类目、商品、店铺等级分层。
3. R16 未完结售后金额是全部 freeze，还是按历史成功退款概率折算。
4. R18 中广告费、达人佣金、平台费用、赔付是否全部具备订单级映射。
5. R20 回款路径不可控时，是直接建议暂停新增，还是先人工核实。
6. R21 重复融资数据由哪个系统提供，是否能做到订单级占用。
7. 原业务系统是否需要接收 `available_to_disburse_reference` 和 `suggested_hold_amount`。
8. 通知运营的渠道、模板、频率和升级策略。
9. 高风险情况下是否允许系统自动冻结未放款额度；如果允许，需要审批和回滚流程。
10. 历史回测样本范围、坏账标签、逾期标签和最终回款标签由谁提供。

## 23. 交流计划

### 23.1 第一场：业务口径会

目标：确认“订单抵押贷款”到底抵押什么、第一还款来源是什么、哪些风险会影响未放款金额。

参会：业务负责人、运营负责人、风控、产品、研发。

必须输出：

1. 首期店铺和订单范围。
2. 订单入池条件。
3. 未放款金额调整边界。
4. 通知和人工核实流程。

### 23.2 第二场：数据字段会

目标：确认数据源能不能支撑 R16、R18、R30 最小闭环。

必须输出：

1. 字段清单。
2. 缺字段替代方案。
3. 数据刷新频率。
4. 授权失效处理。
5. 快照存储和脱敏要求。

### 23.3 第三场：规则和金额台账会

目标：确认每条规则如何影响金额，以及如何防止重复扣减。

必须输出：

1. 规则 owner。
2. `deduct`、`freeze`、`exclude`、`discount` 的定义。
3. R18 和 R19 去重规则。
4. 硬停规则。
5. 参数版本发布流程。

### 23.4 第四场：Agent 和运营报告会

目标：确认运营需要看到什么，以及 Agent 不能说什么。

必须输出：

1. 报告模板。
2. 群聊通知模板。
3. Agent 禁止话术。
4. 人工核实表单。
5. 复跑流程。

### 23.5 第五场：灰度上线会

目标：确认首期灰度范围和验收。

必须输出：

1. 灰度店铺列表。
2. 灰度时间。
3. 每日巡检频率。
4. 告警升级人。
5. 回测和人工复核结果。
6. 是否进入二期规则开发。
