# 风控后台 HTML 版开发实施文档

> 版本：v1.0  
> 日期：2026-07-03  
> 适用范围：电商平台融资风控后台 HTML 版前端开发  
> 目标：基于当前 UI 设计图和产品设计文档，建设一个结构清晰、可接真实接口、可后续迁移到正式后台的专业化 HTML 风控后台。

---

## 1. 开发目标

本后台首版是一个轻量 HTML 风控后台，不是完整 Vue/React 工程，也不是规则配置后台。首版重点完成：

1. 自动巡检结果展示；
2. 手动查询店铺最新巡检结果；
3. 运行记录和失败排查入口；
4. 单店报告详情展示；
5. 后续接入规则引擎、Agent、巡检任务接口时，只替换数据层，不大改页面结构。

前端职责：

- 读取数据；
- 渲染页面；
- 做字段映射；
- 做状态展示；
- 做页面跳转；
- 做只读操作入口；
- 为重新分析、失败明细等后端任务预留入口。

前端不做：

- 风险等级计算；
- 可回款资产计算；
- 放款参考金额计算；
- 规则命中判断；
- Agent 报告生成；
- 审批、放款、停放款、授信调整；
- 自动创建订单同步任务；
- 自动创建报告任务；
- 自动发送通知。

## 2. 总体实施策略

### 2.1 推荐策略

采用：

```text
真实电商基础数据 + mock 风控结果数据
```

也就是：

- 店铺列表、平台枚举优先接真实只读接口；
- 风控巡检结果、风险报告、运行记录先用 mock；
- 页面统一通过 adapter 获取数据；
- 后端风控接口完成后，只替换 adapter，不重写页面。

### 2.2 为什么不全 mock

手动查询页依赖真实店铺列表，否则无法验证：

- 店铺字段是否齐全；
- 平台枚举如何映射；
- 店铺状态、风控绑定状态、第三方店铺 ID 是否真实存在；
- 搜索和分页形态是否贴近真实后台。

### 2.3 为什么不一开始全接真实接口

当前真实电商接口不能直接提供：

- 最新巡检风险等级；
- 可回款资产；
- 放款参考金额；
- 数据状态；
- 报告 ID；
- 报告详情；
- 风险事件；
- 风险证据；
- 运行记录；
- 失败明细。

这些字段属于新风控系统、规则引擎、报告服务、任务记录服务的输出，不能从旧电商接口中硬拼。

## 3. 数据真实性边界

### 3.1 已确认可从现有接口获取的数据

| 页面字段 | 是否真实可取 | 推荐来源 | 说明 |
| --- | --- | --- | --- |
| 店铺 ID | 是 | `GET /admin/merchant.SecuredShop/index` | 店铺主键 |
| 店铺名 | 是 | `shop_name` | 可用于列表展示和搜索 |
| 商户名/商户昵称 | 是 | `merchant_nickname` | 可用于店铺/商户列 |
| 平台类型 | 是 | `platform_type` | 需结合平台枚举映射 |
| 平台名称 | 是 | `GET /admin/risk.RiskReport/riskPlatformType` | 抖音、快手、拼多多等 |
| 第三方店铺 ID | 是 | `third_shop_id` | 平台侧店铺标识 |
| 店铺状态 | 是 | `status` | 需做枚举映射 |
| 风控绑定状态 | 是 | `is_bind_risk_control` | 已绑定/未绑定 |
| 旧风控状态 | 是，但仅作存量字段 | `risk_control_status` | 不能等同新巡检等级 |
| 数据中心状态 | 是 | `offline_status` | 在线/离线/未接入 |
| 店铺指标数组 | 是 | `POST /admin/risk.RiskReport/getRiskReportTaskData` | 只作为诊断或报告补充，不作为手动查询首页主展示 |
| 订单同步时间 | 是 | `POST /risk/risk/getOrderList` | 可用于数据质量或诊断 |
| 支用单列表 | 是 | `GET /admin/ytpay.TradeApplyOrders/index` | 首版不作为主依赖 |

### 3.2 必须 mock 或等待新风控接口的数据

| 页面字段 | 现状 | 原因 |
| --- | --- | --- |
| 巡检批次 | mock | 属于新巡检任务 |
| 最新风险等级 | mock | 规则引擎输出 |
| 可回款资产 | mock | 规则引擎/资产口径计算 |
| 放款参考金额 | mock | 规则引擎计算 |
| 数据状态 | mock | 新风控系统汇总 |
| 报告 ID/报告链接 | mock | 新报告服务输出 |
| 今日结论 | mock | 报告服务/Agent 输出 |
| 风险事件与证据 | mock | 规则引擎输出 |
| 运营核实事项 | mock | 报告服务输出 |
| 运行记录 | mock | 新巡检任务记录 |
| 失败明细 | mock | 新任务日志/错误结构 |
| 重新分析结果 | mock/占位 | 后端任务接口输出 |

### 3.3 不得伪造成真实接口结果的字段

以下字段在 UI 中可以 mock 展示，但代码中必须明确来自 `risk.mock.js` 或 `risk-adapter.js` 的 mock 分支：

- `risk_level`
- `risk_level_name`
- `collectible_asset_amount`
- `loan_reference_amount`
- `data_status`
- `data_snapshot_time`
- `report_id`
- `risk_events`
- `verify_items`
- `audit`
- `task_status`
- `failure_reason`

## 6. JS 分层规范

### 6.1 `core/config.js`

配置：

```js
export const AppConfig = {
  apiBase: 'https://chaolianadmin.test.luoboyoupin.com',
  useMockRisk: true,
  pageSize: 10
};
```

注意：

- 不在代码中写账号密码；
- 不把测试登录凭证提交到文件；
- 接口 token 从登录态或本地调试注入。

### 6.2 `core/request.js`

封装请求：

- GET；
- POST；
- query 参数；
- JSON body；
- 错误处理；
- 登录失效处理；
- 只读保护。

禁止在请求层主动调用写操作接口。

### 6.3 `api/store-api.js`

负责现有电商后台只读接口：

```js
export async function fetchSecuredShopList(params) {}
export async function fetchPlatformTypes() {}
export async function fetchStoreMetrics(params) {}
export async function fetchOrderList(params) {}
```

### 6.4 `api/risk-api.js`

负责未来新风控接口：

```js
export async function fetchLatestInspectionBatch(params) {}
export async function fetchInspectionList(params) {}
export async function fetchLatestStoreInspection(storeId) {}
export async function fetchReportDetail(reportId) {}
export async function fetchRunRecords(params) {}
export async function rerunStoreInspection(storeId) {}
```

首版 `useMockRisk = true` 时，这些函数返回 mock 数据。

### 6.5 `adapters`

adapter 负责把接口数据转换成页面 ViewModel。

例如店铺列表适配：

```js
export function mapSecuredShopToStoreView(row, platformMap) {
  return {
    storeId: row.id,
    shopName: row.shop_name,
    merchantName: row.merchant_nickname,
    platformType: row.platform_type,
    platformName: platformMap[row.platform_type] || '--',
    thirdShopId: row.third_shop_id,
    shopStatus: mapShopStatus(row.status),
    riskBindStatus: mapBindStatus(row.is_bind_risk_control),
    legacyRiskStatus: mapLegacyRiskStatus(row.risk_control_status),
    dataCenterStatus: mapOfflineStatus(row.offline_status)
  };
}
```

页面不得直接依赖原始接口字段。


- 建议动作；
- 主要风险；
- 影响金额；
- 报告状态；
- 通知状态。

```text
pages/manual-query.html
```

## 10. 接口接入顺序

### 第 1 步：完成静态拆分

目标：

- 建立目录；
- 拆出公共 CSS；
- 拆出公共 JS；
- 保持当前 UI 视觉不变。

验收：

- 4 个页面都能独立打开；
- 左侧导航、顶部栏、表格、分页样式统一；
- 页面没有内联大段 CSS/JS。

### 第 2 步：接真实店铺列表和平台枚举

目标：

- 手动查询页使用真实店铺列表；
- 平台名称使用真实平台枚举；
- 店铺字段通过 adapter 映射。

接口：

```http
GET /admin/merchant.SecuredShop/index
GET /admin/risk.RiskReport/riskPlatformType
```

验收：

- 能按关键词查询店铺；
- 能展示店铺名、商户名、平台、第三方店铺 ID；
- 未登录或接口失败时有错误提示；
- 不调用任何写接口。

### 第 3 步：接 mock 风控结果

目标：

- 自动巡检页展示 mock 批次和 mock 结果；
- 手动查询页把真实店铺与 mock 最新巡检结果合并展示；
- 报告详情页展示 mock 报告；
- 运行记录页展示 mock 任务。

## 11. 只读和安全边界

首版禁止自动调用以下接口：

| 接口 | 禁止原因 |
| --- | --- |
| `POST /admin/merchant.SecuredShop/updateOrder` | 创建订单同步任务 |
| `POST /risk/risk/updateOrder` | 创建订单同步任务 |
| `POST /admin/risk.RiskReport/addRiskReportTask` | 创建风控抓取/报告任务 |
| 审核、改状态、驳回、放款确认类接口 | 修改业务状态 |
| 导出接口 | 可能造成敏感数据扩散 |

重新分析在后端接口未确认前不得真实触发。

## 13. 开发注意事项

1. 页面不要使用右侧详情抽屉；
2. 复杂详情用弹框或独立详情页；
3. 筛选项 label 和输入框保持左右结构；
4. 列表字段控制在必要范围内；
5. 不展示“按后端返回顺序展示”等无意义文案；
6. 操作按钮不要贴右边界；
7. 表格列宽总和必须控制在 100%；
8. 金额字段必须标明“参考”口径；
9. 无报告时不伪造报告；
10. 数据缺失不等同于商户风险；
11. 旧 `risk_control_status` 不等同新巡检等级；
12. mock 风控结果不能伪装成真实接口结果；
13. 敏感字段默认不展示或脱敏；
14. 所有跳转都通过 query 参数传递 ID。
