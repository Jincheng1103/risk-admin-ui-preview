# 电商平台融资自动巡检产品设计文档

## 一、产品定位与一句话定义

### 1.1 产品名称

**电商平台融资自动巡检系统**

### 1.2 一句话定义

本产品是面向电商平台融资业务的自动巡检与风险报告系统，用于按配置规则识别影响可回款资产和新增放款参考的风险，并生成通知、报告和留痕，帮助运营、主管和管理层在原业务流程中做放款参考判断。

### 1.3 核心定位

系统只输出风险参考、证据说明、核实事项和建议动作，不在本系统内发起、审批、执行、停止或调整放款。所有放款判断和业务操作仍在原业务系统和原业务流程中完成。

### 1.4 术语定义

| 术语 | 定义 |
| --- | --- |
| 放款前审核参考 | 商家新增放款申请或额度使用前，运营参考系统报告判断是否需要关注风险 |
| 新增放款过程风险复核 | 本文中的“放款中”统一指新增放款过程中的风险复核，不包括逾期后的催收、法务、坏账处置和贷后资产保全流程 |
| 可回款资产 | 融资方有较高把握确认、归集、到账并用于还款扣收的资产金额 |
| 已放款敞口 | 当前已放款但尚未回收的资金敞口，用于风险判断和安全垫计算，不属于可回款资产 |
| 安全垫 | 可回款资产与已放款敞口之间的差额或业务配置口径 |
| 系统自动通知 | 系统按风险等级、通知对象和去重规则自动发送通知 |
| 用户主动发送 | 用户在有权限的前提下，基于已有报告手动触发通知发送 |
| 报告反馈 | 用户对报告是否有用、是否误报的质量反馈，不等同于业务核实结果 |
| 业务核实回流 | 原业务系统或人工处置结果回传到本系统后形成的核实结果 |

### 1.5 核心分工

```text
规则引擎：识别风险事件，输出风险等级、影响金额、建议动作、通知策略和证据。
Agent：基于规则引擎输出、报告模板和提示词生成通知摘要、完整报告、风险解释和核实事项。
通知系统：按通知策略将需要关注的巡检结论推送到通知群或短信。
报告页：承载完整风险分析、规则证据、核实事项和历史变化。
用户端工作台：供运营、主管和管理层查看报告、查询店铺、查看总览和反馈报告。
管理后台：供管理员和实施人员配置规则、资产口径、报告模板、通知对象、巡检任务和权限。
```

## 二、业务背景与核心问题

电商平台商家销售后，平台货款通常不会马上结算给商家。融资业务会根据商家的订单、待结算货款、经营表现和风险情况，提前按一定比例放款，帮助商家缓解账期压力。

放款安全的关键不是订单数量，也不是 GMV，而是融资方最终能够确认、归集、到账并用于还款扣收的可回款资产。GMV、订单量等经营指标可以作为风险信号，但不能直接等同于可回款资产。

可回款资产口径可能因平台、业务规则、风险偏好和融资方案不同而变化，因此系统支持通过规则引擎和字段映射配置资产口径，不在产品层固定唯一计算公式。

系统每天要回答以下问题：

1. 哪些店铺的可回款资产正在缩水；
2. 哪些店铺的可回款资产可能延迟到账；
3. 哪些订单、商品、售后、履约、结算或账户风险会影响放款安全；
4. 哪些店铺出现放款前审核参考或新增放款过程风险升高；
5. 哪些店铺不适合继续按原参考比例新增放款。

## 三、数据来源与核心资产口径

### 3.1 数据来源

系统巡检基于业务系统同步的数据快照。数据分为业务数据和系统配置数据。

业务数据包括：

1. 店铺基础信息；
2. 企业主体信息；
3. 订单数据；
4. 商品数据；
5. 售后退款数据；
6. 发货履约数据；
7. 结算账单数据；
8. 回款账户数据；
9. 放款记录；
10. 还款扣收记录。

系统配置数据包括：

1. 运营负责人绑定；
2. 通知对象配置；
3. 规则配置；
4. 资产口径配置；
5. 报告模板配置；
6. Agent 提示词配置；
7. 巡检计划配置；
8. 用户角色和数据范围配置。

每次巡检必须记录数据快照时间。关键数据超过配置的新鲜度阈值时，巡检结果和报告必须标记数据可能滞后。

### 3.2 数据新鲜度

不同数据源可以配置不同的新鲜度阈值。

| 数据类型 | 示例阈值 | 过期后的处理 |
| --- | --- | --- |
| 订单、售后、履约 | 按小时或按日配置 | 标记数据过期，并提示风险结论可能滞后 |
| 结算账单、回款账户 | 按账单周期或同步周期配置 | 标记结算或账户数据过期，必要时降低结论确定性 |
| 放款、还款扣收 | 按业务系统同步周期配置 | 标记敞口或扣收数据可能滞后 |
| 负责人、通知对象、权限 | 按配置更新时间记录 | 配置缺失时按异常流程处理 |

关键数据缺失或过期时，系统仍可生成巡检结果，但必须降低结论确定性并在报告顶部提示。若缺失字段导致风险等级、影响金额或建议动作无法计算，规则引擎应输出“无法判断”或配置的兜底等级，并说明不能计算的字段。是否通知由规则配置决定。

### 3.3 巡检范围与频率

自动巡检范围默认是已纳入融资业务且状态有效的店铺。不在融资业务范围内、已停用、未完成基础配置或被管理员排除的店铺不进入自动巡检，但可按权限手动查询。

手动查询非自动巡检范围内的店铺时，系统应展示无法自动分析的原因；如果缺少必要业务数据或基础配置，不生成完整风险结论。

系统默认每天定时巡检一次，具体执行时间由管理员配置。系统支持手动触发单店重新分析；是否支持多次定时巡检由巡检计划配置决定。

### 3.4 可回款资产口径

系统支持管理员通过规则引擎和字段映射配置可回款资产口径。报告模板和 Agent 提示词只负责控制口径的展示名称、解释方式和报告表达，不参与公式计算。

可回款资产口径由基础资产、扣减项和风险冻结项共同构成；已放款敞口不属于可回款资产，但应作为风险判断和安全垫计算的对照字段。

可配置字段可分为四类：

| 类型 | 示例 | 说明 |
| --- | --- | --- |
| 基础资产 | 待结算货款、已确认应收 | 可能用于还款扣收的基础金额 |
| 扣减项 | 已发生退款、预计退款、售后扣减、无效订单、取消订单、异常订单扣减、未发货或履约异常扣减 | 可能降低可回款资产质量的金额 |
| 风险冻结项 | 货款冻结、结算暂停、账户异常金额 | 可能导致资产无法按期归集或扣收的金额 |
| 对照项 | 已放款敞口、安全垫 | 用于判断风险暴露和缓冲空间，不直接等同于可回款资产 |

### 3.5 规则引擎资产字段输出

规则引擎每次必须输出本次使用的资产口径名称、口径版本、基础资产金额、扣减项金额、风险冻结项金额、可回款资产、已放款敞口、安全垫、影响金额和无法估算原因。具体计算公式由配置决定，但字段含义和版本必须可追溯。

| 输出字段 | 说明 |
| --- | --- |
| 资产口径名称 | 本次报告使用的资产口径展示名称 |
| 资产口径版本 | 本次计算使用的配置版本 |
| 基础资产金额 | 规则配置中纳入基础资产的金额汇总 |
| 扣减项金额 | 已发生或预计影响资产质量的扣减金额 |
| 风险冻结项金额 | 可能导致资产无法按期归集或扣收的金额 |
| 可回款资产 | 按配置口径计算后的可参考资产 |
| 已放款敞口 | 当前已放款但尚未回收的资金敞口 |
| 安全垫 | 可回款资产与已放款敞口之间的差额或业务配置口径 |
| 影响金额 | 本次风险事件预计影响的金额 |
| 无法估算原因 | 数据缺失、口径冲突或证据不足导致无法估算时填写 |

### 3.6 口径配置原则

1. 资产口径由管理员在规则引擎和字段映射中配置，系统不内置唯一公式；
2. 规则引擎负责按配置计算或输出相关字段；
3. Agent 只能解释已配置口径，不得自行创造计算公式；
4. 报告必须展示本次使用的关键口径名称、核心字段、配置版本和规则证据；
5. 如果数据缺失或口径配置冲突，报告必须明确提示，不允许 Agent 猜测补全。

## 四、产品目标与边界

### 4.1 产品目标

本产品解决三个核心问题：

1. 运营不用每天从大量数据中手工找风险。系统自动巡检纳入融资业务的有效店铺，把异常店铺挑出来。
2. 运营看到通知后能快速判断是否需要关注。通知提供店铺、风险等级、核心原因、建议动作和报告链接。
3. 运营打开报告后能看懂风险原因。报告说明风险来源、影响金额、关联对象、证据、风险链路和放款参考建议。

### 4.2 产品范围

本产品服务于放款前审核参考和新增放款过程中的风险复核。

本产品只做：

1. 自动巡检；
2. 手动查询；
3. 单店重新分析；
4. 风险事件识别；
5. 风险等级计算；
6. 影响金额估算；
7. 放款参考建议；
8. 核实事项生成；
9. 系统自动通知；
10. 用户主动发送通知；
11. 风险报告；
12. 每日总览；
13. 历史留痕；
14. 基础角色权限和数据范围权限；
15. 基础配置和运行排查。

本产品不做：

1. 在本系统内发起、审批、执行或变更放款；
2. 自动执行放款；
3. 自动停止放款；
4. 自动调整额度；
5. 自动生成处置工单；
6. 催收管理、法务管理、逾期处置或坏账管理；
7. CRM、商家经营跟进、销售线索、客户拜访、客户生命周期运营等能力；
8. 复杂审批流、字段级权限、跨组织授权和细粒度权限编排系统。

### 4.3 决策边界

系统输出的是风险参考，不是最终决策。

```text
系统可以提示：该店铺风险升高，建议谨慎放款或建议暂停新增。
系统不能执行：自动放款、自动停止放款、自动调整额度、自动生成处置工单。
最终判断由运营或管理人员在原有业务系统和业务流程中完成。
```

## 五、用户角色与使用入口

### 5.1 运营人员

运营人员是主要使用者。运营主要通过通知群、短信、报告页和用户端工作台使用系统。

运营关注：

1. 今天哪些店铺有风险；
2. 哪些店铺建议谨慎放款；
3. 哪些店铺建议暂停新增或人工核实；
4. 为什么这个店铺有风险；
5. 哪些订单、商品、售后、履约、结算或账户导致风险；
6. 当前报告能否作为放款参考。

运营不需要频繁进入管理后台；只在查看完整报告、手动查询店铺、回看历史报告或触发重新分析时进入用户端工作台或报告页。

主要入口：

1. 通知群；
2. 短信链接；
3. 报告页；
4. 店铺查询页；
5. 报告列表。

### 5.2 主管 / 管理层

主管和管理层关注整体风险盘面和运营管理。

主要关注：

1. 今日高风险店铺数量；
2. 严重风险店铺数量；
3. 风险金额规模；
4. 主要风险来源；
5. 各运营负责店铺的风险情况；
6. 近期风险变化；
7. 系统巡检、报告和通知是否正常。

主要入口：

1. 首页概览；
2. 每日总览报告；
3. 报告列表；
4. 店铺查询页；
5. 风险汇总；
6. 通知记录；
7. 报告反馈。

其中风险汇总归属首页概览或报告列表，报告反馈归属报告详情或报告列表，页面归属以第十一章为准。

### 5.3 管理员 / 实施人员

管理员和实施人员负责系统配置、运行维护和问题排查。

主要关注：

1. 巡检任务是否正常；
2. 规则是否启用；
3. 资产口径是否配置正确；
4. Agent 提示词和报告模板是否正确；
5. 报告是否生成成功；
6. 通知是否发送成功；
7. 店铺是否绑定正确负责人；
8. 权限和通知对象是否配置正确；
9. 某店铺为什么没有生成报告；
10. 某次通知为什么没有成功发送。

主要入口：

1. 管理后台；
2. 规则配置；
3. 报告模板配置；
4. 系统设置；
5. 店铺查询页；
6. 巡检任务记录；
7. 运行日志；
8. 异常监控。

其中巡检任务记录、运行日志和异常监控归属系统设置或运行管理二级页面，页面归属以第十一章为准。

## 六、核心流程

### 6.1 自动巡检流程

```text
定时巡检任务启动
  ↓
筛选纳入融资业务且状态有效的店铺
  ↓
按店铺生成巡检快照并记录数据快照时间
  ↓
规则引擎读取资产口径配置、风险规则、等级规则和通知规则
  ↓
规则引擎输出结构化风险事件、风险等级、影响金额、建议动作、通知策略和证据
  ↓
系统判断是否生成单店报告、是否进入每日总览、是否触发通知
  ↓
Agent 基于规则事件、证据、报告模板和提示词生成报告及通知摘要
  ↓
通知系统按策略、接收人和去重规则推送通知群或短信
  ↓
用户点击链接查看完整报告
  ↓
系统记录巡检、报告、通知、查看和反馈
```

通知失败不影响已生成报告的查看；Agent 失败时保留规则引擎结果，并按异常降级流程生成简化报告或标记失败。

### 6.2 报告生成和通知默认规则

每日总览覆盖本次巡检范围内所有成功生成巡检结果的店铺。正常和关注风险默认只进入每日总览，不单独外发通知；高风险和严重风险同时进入每日总览、生成单店风险报告，并按通知策略外发。

默认规则：

1. 高风险和严重风险生成单店风险报告，并按通知策略提醒；
2. 正常和关注风险进入每日总览，不单独外发通知；
3. 管理员可配置关注风险是否生成单店报告；
4. 生成单店报告和外发通知是两个独立开关，关注风险即使生成单店报告，也不默认外发通知；
5. 用户触发重新分析后，系统生成新的分析结果和报告版本；
6. 如果关键数据缺失，报告必须标记缺失字段和影响，不得隐藏缺失状态。

### 6.3 触发方式矩阵

| 触发方式 | 是否生成新结果 | 是否生成单店报告 | 是否外发通知 | 备注 |
| --- | --- | --- | --- | --- |
| 定时巡检 | 是 | 高风险、严重风险默认生成；关注可配置 | 按风险等级策略 | 所有结果进入每日总览 |
| 手动查询 | 否 | 否，优先展示已有报告 | 否 | 无最新报告时展示最新巡检摘要，并提示可重新分析 |
| 重新分析 | 是 | 是，生成新版本 | 默认否 | 严重风险可配置触发通知 |
| 用户主动发送 | 否 | 使用已有报告 | 是 | 需权限、通知策略、去重规则和发送原因 |

最新报告指同一店铺在当前有效数据快照或管理员配置的有效时间窗口内生成的最近一份成功报告。若没有最新报告，系统展示最新巡检摘要，并提示用户可触发重新分析。

### 6.4 手动查询流程

```text
用户进入店铺查询页
  ↓
搜索店铺名称 / 店铺 ID / 企业主体 / 运营负责人
  ↓
系统按用户数据范围展示最新风险摘要
  ↓
用户查看完整报告或历史报告
  ↓
必要时触发重新分析
  ↓
系统基于最新数据生成新的分析结果和报告版本
  ↓
默认仅当前用户查看，不自动外发通知
```

手动查询典型场景：

1. 商家临时申请放款，运营手动查询店铺风险；
2. 主管抽查某个高风险店铺；
3. 运营收到商家解释后，重新分析相关店铺；
4. 管理员排查某店铺为什么没有生成报告。

### 6.5 重新分析与用户主动发送

手动重新分析默认不外发。若命中严重风险且规则配置为“允许手动分析触发通知”，系统按严重风险通知策略发送；同一风险事件在去重周期内不重复外发，除非用户主动发送并记录原因。

用户主动发送通知时，必须满足角色权限、通知策略和去重规则，并记录发送人、发送对象、发送原因、发送时间和发送结果。

### 6.6 异常降级流程

当数据缺失、数据过期、规则失败、Agent 失败、通知失败或配置冲突时，系统必须保留可用结果、标记失败原因、记录影响范围，并按配置决定是否生成简化报告、是否通知以及是否阻断相关流程。

处理原则：

1. 数据缺失：在巡检结果和报告中标记缺失字段、数据时间和可能影响；
2. 数据过期：标记数据新鲜度异常，并提示管理员检查同步；
3. 规则失败：记录失败规则、失败原因、影响店铺和影响范围；
4. Agent 失败：保留规则引擎输出，使用固定兜底模板生成简化报告；简化报告必须标记“AI 报告生成失败，以下为规则引擎结果摘要”；
5. 通知失败：记录失败渠道、失败原因、发送对象和重试状态；
6. 配置冲突：按阻断型、降级型和仅提示型处理；
7. 部分失败：保留成功结果，同时标记失败店铺和失败原因。

配置冲突分级：

| 类型 | 处理方式 | 示例 |
| --- | --- | --- |
| 阻断型 | 停止相关店铺、相关规则、报告生成或通知发送 | 资产口径缺少核心字段、风险等级规则无有效版本 |
| 降级型 | 保留可计算结果，生成简化报告或降低结论确定性 | 部分证据字段缺失、部分通知对象缺失 |
| 仅提示型 | 不阻断结果生成，在后台和报告中提示管理员修正 | 展示文案缺失、非核心模板字段缺失 |

所有配置冲突必须记录配置项、版本、影响范围和处理结果。

## 七、规则引擎与 Agent 分工

### 7.1 基本原则

规则引擎是风险判断、风险等级、影响金额、建议动作、通知策略和证据的唯一来源。Agent 只基于规则引擎输出、报告模板和提示词生成可读表达，不得新增事实、改变等级、改变金额、改变建议动作或创造资产计算公式。

### 7.2 规则引擎负责

规则引擎负责稳定、可追溯、可解释的判断。

规则引擎负责：

1. 风险事件识别；
2. 可回款资产口径计算或字段输出；
3. 风险等级计算；
4. 影响金额估算；
5. 建议动作输出；
6. 通知策略判断；
7. 是否进入报告判断；
8. 数据缺失、证据不足和口径冲突标记；
9. 输出规则证据和配置版本。

### 7.3 Agent 负责

Agent 负责把结构化风险事件写成用户能看懂的通知和报告。

Agent 负责：

1. 生成简短通知摘要；
2. 生成完整风险报告；
3. 归纳主要风险原因；
4. 解释风险链路；
5. 按规则引擎结果表达放款参考建议；
6. 列出运营需要核实的问题；
7. 展示规则引擎或数据校验给出的数据不足、证据不足或口径冲突。

Agent 可以调整语言表达、归纳顺序和可读性，但不得新增风险事件、改变风险等级、改变影响金额、改变建议动作或补充规则引擎未给出的事实。

### 7.4 Agent 禁止事项

Agent 不得：

1. 基于原始数据重新判断风险等级；
2. 修改规则引擎给出的影响金额；
3. 修改规则引擎给出的建议动作；
4. 自行创造可回款资产计算公式；
5. 在证据不足时编造原因；
6. 输出自动放款、自动停止放款或自动调整额度的执行性指令；
7. 代替运营或管理人员做最终放款决策。

### 7.5 示例

以下示例只说明表达方式，不代表固定规则。

```text
规则事件：
售后退款金额异常上升，高风险，预计影响金额 {金额}，建议动作：谨慎放款。

Agent 报告：
该店铺今日风险升高，主要因为售后退款集中上升。根据规则引擎输出，本次风险预计影响金额 {金额}，建议在核实售后原因前谨慎放款。运营需重点确认退款是否继续扩大，以及相关订单是否仍可作为放款参考。
```

## 八、风险等级、建议动作与证据标准

### 8.1 风险等级

风险等级统一为四档。系统内部枚举和等级顺序固定，首期不支持新增等级或改变等级层级。管理员可以配置各等级触发条件、阈值、展示文案和通知策略。

| 系统值 | 展示名 | 含义 |
| --- | --- | --- |
| normal | 正常 | 未发现明显异常 |
| watch | 关注 | 有轻微异常，需要观察 |
| high | 高风险 | 已影响可回款资产质量或放款安全 |
| critical | 严重风险 | 可能导致回款失控、资产明显缩水或重大放款敞口风险 |

### 8.2 影响金额

影响金额由规则引擎根据管理员配置的业务口径输出。报告必须展示本次规则输出的关键金额字段；如果部分字段缺失，应展示缺失原因和影响。

报告中应区分以下金额字段：

| 字段 | 含义 |
| --- | --- |
| 待结算货款 | 平台或业务系统提供的待结算金额 |
| 预计风险扣减 | 规则判断可能不可作为放款参考的金额 |
| 可回款资产 | 按配置口径计算后的可参考资产 |
| 已放款敞口 | 当前已放款但尚未回收的资金敞口 |
| 安全垫 | 可回款资产与已放款敞口之间的差额或业务配置口径 |

字段名称、计算方式和展示方式由管理员配置，但配置版本必须留痕。

### 8.3 建议动作

建议动作统一为放款参考动作，遵循 4.3 决策边界，不在本系统内强制执行。

| 系统值 | 展示名 | 含义 |
| --- | --- | --- |
| observe | 正常观察 | 当前无明显风险，可按原流程判断 |
| cautious | 谨慎放款 | 存在风险，放款前需关注报告原因 |
| reduce_ratio | 建议降低参考比例 | 建议按规则配置的折扣或扣减后资产判断 |
| pause_new | 建议暂停新增 | 建议暂不新增放款，待人工核实后再判断 |
| manual_check | 人工核实 | 需要运营核实商家、订单、商品、售后、履约、结算或账户情况 |

### 8.4 风险等级与建议动作默认映射

以下为默认映射，最终建议动作以规则引擎输出为准。

| 风险等级 | 默认建议动作 | 说明 |
| --- | --- | --- |
| 正常 | 正常观察 | 无明显异常，按原流程判断 |
| 关注 | 正常观察 / 人工核实 | 轻微异常，视规则配置决定是否核实 |
| 高风险 | 谨慎放款 / 建议降低参考比例 | 已影响资产质量或放款安全 |
| 严重风险 | 建议暂停新增 / 人工核实 | 存在回款失控或重大敞口风险 |

### 8.5 核实事项

核实事项用于告诉运营“具体要看什么”，不等同于建议动作。

示例：

1. 核实售后退款是否继续扩大；
2. 核实异常商品是否存在质量或履约问题；
3. 核实结算账户是否发生变更；
4. 核实未发货订单是否仍能正常履约；
5. 核实平台货款是否存在冻结或结算暂停。

### 8.6 证据标准

每个风险事件至少应包含：

1. 命中规则；
2. 关键指标；
3. 对比基准；
4. 数据时间；
5. 关联对象；
6. 影响金额口径；
7. 规则配置版本。

缺少关键证据时，风险事件可以进入报告，但必须标记“证据不足”或“数据缺失”。证据不足事件的风险等级、建议动作和通知策略仍由规则引擎输出；如果规则引擎无法判断，应按配置输出兜底等级或“无法判断”状态，并说明原因。

## 九、通知与报告

### 9.1 通知和报告的关系

通知只解决一个问题：提醒用户有风险需要查看。

报告解决完整问题：为什么有风险、证据是什么、影响多少、建议如何作为放款参考、运营需要核实什么。

所以通知必须短，报告必须完整。

### 9.2 通知内容原则

通知内容包含：

1. 一句话结论；
2. 店铺名称；
3. 风险等级；
4. 核心原因；
5. 影响金额或无法估算说明；
6. 建议动作；
7. 报告链接。

### 9.3 风险等级默认通知策略

| 风险等级 | 默认通知策略 |
| --- | --- |
| 正常 | 不单独通知，进入每日总览 |
| 关注 | 不单独通知，进入每日总览；是否生成单店报告可配置 |
| 高风险 | 生成单店报告，推送通知群 |
| 严重风险 | 生成单店报告，推送通知群，并发送短信 |

管理层或主管可以接收每日总览通知，默认通过通知群发送。

### 9.4 触发方式通知策略

| 触发方式 | 默认通知策略 |
| --- | --- |
| 定时巡检 | 按风险等级默认通知策略发送 |
| 手动查询 | 不生成新报告，不自动外发通知 |
| 重新分析 | 默认不外发；严重风险且配置允许时按严重风险策略发送 |
| 用户主动发送 | 按权限、通知策略和去重规则发送，并记录发送原因 |

### 9.5 通知接收人和兜底策略

1. 单店高风险通知默认发送给该店铺运营负责人所在通知群；
2. 严重风险在通知群基础上短信发送给配置的责任人；
3. 每日总览默认发送给主管或管理层通知群；
4. 通知对象由店铺负责人绑定、通知群配置和短信联系人配置共同决定；
5. 店铺未绑定运营负责人时，发送到管理员配置的兜底通知群；
6. 严重风险缺少短信联系人时，仍发送通知群，并在通知记录和后台标记短信联系人缺失；
7. 通知群配置缺失时，不阻断报告生成，但必须记录通知失败原因并提示管理员修正。

### 9.6 通知去重和升级

通知去重以风险事件指纹为准。默认指纹由店铺 ID、风险类型、命中规则 ID、关联对象、风险统计窗口和风险等级组成。仅风险描述相同但关联对象或规则不同的事件，不应被简单合并去重。

同一风险事件指纹在配置的去重周期内只外发一次通知。风险等级从高风险升级为严重风险时允许再次通知。重新分析命中相同风险时默认不重复外发，除非用户主动发送并记录原因。

### 9.7 通知模板

以下模板为示例，具体字段、文案和变量由管理员配置。

#### 单店高风险通知

```text
【电商平台融资预警】
店铺：{店铺名称}
等级：{风险等级}
原因：{核心原因}
影响：{影响金额或无法估算说明}
建议：{建议动作和核实事项}
报告：{链接}
```

#### 严重风险通知

```text
【电商平台融资预警】
店铺：{店铺名称}
等级：严重风险
原因：{核心原因}
影响：{影响金额或无法估算说明}
建议：{建议动作和核实事项}
报告：{链接}
```

#### 每日总览通知

```text
【电商平台融资巡检总览】
今日巡检 {巡检店铺数} 家店铺，高风险 {高风险店铺数} 家，严重风险 {严重风险店铺数} 家。
主要风险：{主要风险类型}
查看每日总览：{链接}
```

### 9.8 单店风险报告

单店风险报告回答运营最关心的问题：

1. 这个店铺今天风险高不高；
2. 为什么风险高；
3. 影响哪些金额或资产口径；
4. 影响哪些订单、商品、售后、履约、结算或账户；
5. 建议如何做放款参考；
6. 运营应该重点核实什么。

报告结构：

```text
一、今日结论
二、风险等级与放款参考
三、资产口径变化
四、主要风险事件与证据
五、影响对象
六、风险链路
七、运营核实事项
八、历史变化
九、数据缺失或证据不足说明
```

### 9.9 每日总览报告

每日总览报告给主管和管理层查看整体风险，覆盖本次巡检范围内所有成功生成巡检结果的店铺。

报告结构：

```text
一、今日巡检概况
二、风险等级分布
三、重点风险店铺
四、主要风险类型
五、预计影响金额
六、较昨日变化
七、需重点关注事项
八、系统运行状态
```

### 9.10 报告反馈

运营可标记报告是否有用、是否误报并添加备注。报告反馈只用于评价报告质量，不改变本次风险等级和建议动作。

人工核实结果属于原业务系统或人工处置结果，只有回流到本系统后，才用于统计规则有效性和风险参考效果。

## 十二、权限与安全

### 12.1 权限原则

本产品需要基础角色权限和数据范围权限，用于控制报告、店铺、通知记录和配置入口的访问；不建设复杂审批流、字段级权限、跨组织授权和细粒度权限编排系统。

### 12.2 角色动作矩阵

| 动作 | 运营 | 主管 / 管理层 | 管理员 / 实施人员 |
| --- | --- | --- | --- |
| 查看自己负责店铺报告 | 支持 | 支持管辖范围 | 支持全部或授权范围 |
| 查看每日总览 | 按权限 | 支持 | 支持 |
| 手动查询店铺 | 自己负责店铺 | 管辖范围 | 全部或授权范围 |
| 触发重新分析 | 自己负责店铺 | 管辖范围可配置 | 支持 |
| 用户主动发送通知 | 按权限可配置 | 按权限可配置 | 支持 |
| 查看通知记录 | 自己负责店铺 | 管辖范围 | 全部或授权范围 |
| 标记报告反馈 | 支持 | 支持 | 可查看 |
| 配置规则和资产口径 | 不支持 | 默认不支持 | 支持 |
| 配置模板和提示词 | 不支持 | 默认不支持 | 支持 |
| 管理用户和权限 | 不支持 | 默认不支持 | 支持 |

### 12.3 运营权限

运营可以：

1. 接收自己负责店铺相关通知；
2. 查看自己负责店铺的报告；
3. 手动查询自己负责的店铺；
4. 查看自己负责店铺的历史报告；
5. 触发自己负责店铺的重新分析；
6. 按权限主动发送通知；
7. 标记报告是否有用；
8. 添加简单备注。

运营不能：

1. 修改规则；
2. 修改资产口径；
3. 修改 Agent 提示词；
4. 修改报告模板；
5. 修改通知配置；
6. 管理用户。

### 12.4 主管 / 管理层权限

主管 / 管理层可以：

1. 接收每日总览；
2. 查看管辖范围内的店铺报告；
3. 查询管辖范围内店铺风险和历史报告；
4. 查看风险汇总；
5. 查看管辖范围内通知记录；
6. 查看报告反馈；
7. 查看首页概览和系统运行概况。

主管 / 管理层一般不配置系统。

### 12.5 管理员 / 实施人员权限

管理员 / 实施人员可以：

1. 配置规则；
2. 配置资产口径；
3. 配置报告模板；
4. 配置 Agent 提示词；
5. 配置通知群；
6. 配置短信联系人；
7. 配置巡检任务；
8. 管理用户权限；
9. 查看全部店铺巡检结果；
10. 查看全部报告和通知记录；
11. 查看运行日志；
12. 排查报告生成和通知失败问题。

### 12.6 报告链接权限

报告链接不是免登录公开链接。用户打开链接时，系统先校验登录态或一次性 token，再校验用户角色和数据范围。校验通过后展示报告；校验失败时拒绝访问并记录访问人、链接来源、报告 ID、失败原因和访问时间。

1. 运营只能查看自己负责店铺的报告；
2. 主管和管理层只能查看管辖范围内报告；
3. 管理员 / 实施人员可按权限查看全部报告及运行记录；
4. 外部转发链接不得绕过权限校验；
5. 短信链接应具备有效期，并在访问报告时进行身份或一次性 token 校验。

### 12.7 主动发送通知权限

用户主动发送通知必须满足：

1. 用户对报告对象有查看权限；
2. 用户角色具备主动发送权限；
3. 报告处于可发送状态；
4. 通知对象可解析；
5. 未违反去重规则，或用户具备再次发送权限并填写原因。

### 12.8 敏感信息与脱敏

涉及法人 / 实控人、回款账户、手机号、短信联系人等敏感信息时，页面和报告默认脱敏展示；只有具备相应权限的用户可以查看完整信息。

敏感信息查看行为需要留痕，至少记录查看人、查看时间、查看字段、查看对象、访问入口和权限校验结果。

## 十三、运行留痕与异常处理

### 13.1 留痕范围

每次巡检都要保留：

1. 巡检时间；
2. 店铺对象；
3. 数据快照时间；
4. 配置口径版本；
5. 命中规则；
6. 风险事件；
7. 规则引擎输入摘要；
8. 规则引擎输出；
9. Agent 输入；
10. Agent 输出；
11. 报告正文；
12. 通知内容；
13. 通知发送状态；
14. 用户查看记录；
15. 用户反馈；
16. 失败、部分失败和人工重试记录；
17. 版本信息。

留痕用于支持业务追溯、问题排查和审计核验，确保系统能够回答：

> 当天为什么通知这个店铺？报告根据什么生成？使用了哪版规则、资产口径、模板和提示词？用户有没有打开？运营是否认为有用？

### 13.2 留痕要求

1. 留痕记录应保留配置版本、输入摘要、输出结果、执行状态和失败原因；
2. 涉及敏感信息的日志应脱敏；
3. Agent 输入和 Agent 输出可能包含敏感信息，必须按日志权限和脱敏规则控制访问；
4. 留痕保留周期由管理员配置，默认不少于业务追溯所需周期；
5. 通知发送、重试、主动发送、权限拒绝访问都必须记录；
6. 简化报告和 Agent 失败状态必须在报告和运行日志中标记。

### 13.3 异常处理

| 异常类型 | 处理方式 |
| --- | --- |
| 数据缺失 | 标记缺失字段、数据时间和可能影响 |
| 数据过期 | 标记数据新鲜度异常，并提示管理员检查同步 |
| 规则失败 | 记录失败规则、失败原因、影响店铺和影响范围 |
| Agent 失败 | 使用固定兜底模板生成简化报告，并标记失败状态 |
| 通知失败 | 记录失败渠道、失败对象、失败原因和重试状态 |
| 阻断型配置冲突 | 阻断相关巡检、报告生成或通知发送，并提示管理员修正 |
| 降级型配置冲突 | 保留可计算结果，生成简化报告或降低结论确定性 |
| 仅提示型配置冲突 | 不阻断结果生成，在后台和报告中提示管理员修正 |
| 权限校验失败 | 拒绝访问并记录访问人、报告 ID 和失败原因 |

## 十四、产品指标

### 14.1 指标边界

系统内指标只统计巡检、报告、通知、查看和反馈行为。放款比例调整、暂停新增、人工核实结果等业务动作，只有在原业务系统回流结果后才纳入统计。

“应巡检店铺数”“应生成报告数”“应发送通知次数”等分母按本次任务范围、规则配置和通知策略计算，并记录统计周期。

### 14.2 使用指标

| 指标 | 口径 |
| --- | --- |
| 巡检完成率 | 成功完成巡检店铺数 / 应巡检店铺数 |
| 报告生成成功率 | 成功生成报告数 / 应生成报告数 |
| 通知发送成功率 | 成功发送通知次数 / 应发送通知次数 |
| 高风险报告查看率 | 已查看高风险报告数 / 已通知高风险报告数 |
| 严重风险短信发送成功率 | 严重风险短信发送成功次数 / 应发送严重风险短信次数 |
| 手动查询次数 | 用户主动查询店铺风险的次数 |

如果短信供应商提供到达回执，可在短信发送成功率之外增加短信到达率。

### 14.3 报告质量指标

| 指标 | 口径 |
| --- | --- |
| 报告反馈率 | 有反馈报告数 / 已查看报告数 |
| 有用反馈率 | 标记有用报告数 / 有反馈报告数 |
| 误报标记率 | 标记误报报告数 / 有反馈报告数 |
| 简化报告占比 | 简化报告数 / 应生成报告数 |
| 数据缺失报告占比 | 标记数据缺失报告数 / 已生成报告数 |

### 14.4 外部回流后可统计指标

以下指标需要原业务系统回流动作结果后才能纳入系统统计，属于外部回流后可统计指标，不代表本系统执行放款调整或处置动作。

| 指标 | 口径 |
| --- | --- |
| 高风险店铺人工核实率 | 已回流人工核实结果的高风险店铺数 / 高风险店铺数 |
| 严重风险店铺及时查看率 | 配置时限内被查看的严重风险报告数 / 已通知严重风险报告数 |
| 风险预警后放款参考比例调整次数 | 原业务系统回流的放款参考比例调整次数 |
| 风险预警后暂停新增次数 | 原业务系统回流的暂停新增次数 |
| 规则命中后人工确认有效率 | 人工确认为有效的规则命中次数 / 已回流人工确认的规则命中次数 |

## 十五、后续规划与外部依赖

### 15.1 后续规划

以下能力不属于首期核心闭环，可在后续版本中建设：

1. 报告反馈处理状态、处理人、处理结论和关联规则调整记录；
2. 业务核实回流结果的自动同步和统计；
3. 更细粒度的字段级权限和跨组织授权；
4. 更复杂的审批流或处置工单能力；
5. 更多通知渠道和通知到达回执。

### 15.2 外部依赖

本系统依赖外部业务系统提供：

1. 店铺、订单、商品、售后、履约、结算、账户、放款和还款扣收数据；
2. 原业务系统中的最终放款判断和业务操作结果；
3. 放款参考比例调整、暂停新增、人工核实等回流结果；
4. 短信供应商的发送结果或到达回执；
5. 登录态、组织架构、用户身份和数据范围信息。
